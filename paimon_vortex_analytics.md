*Apache Paimon + Vortex: Архитектура сверхбыстрого Real-time Lakehouse*

Интеграция современного формата хранения потоковых данных (Apache Paimon) с высокопроизводительным файловым форматом (Vortex) представляет собой передовое направление для построения аналитических систем нового поколения. Эта связка объединяет способность Paimon «переваривать» огромные потоки мутаций (upserts) с невероятной скоростью чтения векторизованных данных, которую предлагает Vortex.
Такое архитектурное решение отлично подходит для глубокого исследования и создания сравнительных бенчмарков производительности, так как оно бросает вызов традиционным связкам с Parquet или ORC.

*Какие проблемы решает эта связка?*

Традиционно Apache Paimon использует Parquet, ORC или Avro для хранения данных на диске. Несмотря на их эффективность в сжатии, при чтении возникают следующие проблемы:
 * CPU-bottleneck при чтении: Десериализация и распаковка сложных словарей Parquet/ORC потребляют значительные ресурсы процессора.
 * Неоптимальность для современных движков: Аналитические in-memory движки (на базе Apache Arrow, DuckDB) вынуждены тратить время на конвертацию данных из Parquet в свои внутренние векторизованные форматы.
 
Использование Vortex решает эти задачи: он позволяет передавать данные из файлового хранилища Paimon в аналитический движок практически без накладных расходов на CPU (zero-copy) и с максимальным использованием SIMD-инструкций процессора.

*На каком этапе используется Vortex в Paimon?*

В архитектуре Paimon формат Vortex должен применяться исключительно на уровне физического хранения данных (Storage Layer).

Архитектура Paimon базируется на LSM-дереве (Log-Structured Merge-tree). Vortex вступает в игру в следующих точках:
 * Flush (Сброс из памяти): Когда данные накапливаются в оперативной памяти (MemTable) и достигают порога, они сбрасываются на диск (в объектное хранилище или HDFS) в виде неизменяемых файлов (Data Files). Именно эти файлы будут записываться в формате Vortex.
 * Compaction (Уплотнение): Фоновые процессы Paimon читают множество мелких файлов Vortex, сливают их (merging), разрешают конфликты версий строк и записывают новые, более крупные файлы Vortex на нижних уровнях LSM-дерева.

При этом журналы упреждающей записи (WAL / Changelog files), если они используются для потокового чтения, могут оставаться в более легковесных строковых форматах, так как там важнее скорость последовательной записи, а не векторизованного чтения.

*Архитектурная реализация*

Чтобы "подружить" эти технологии, реализация должна затрагивать абстракции файлового ввода-вывода внутри Paimon. Архитектурно это выглядит так:
 * Интерфейс FileFormat: В исходном коде Paimon необходимо реализовать новый класс (например, VortexFileFormat), который наследует базовые интерфейсы Paimon для работы с файлами.
 * Reader и Writer: * VortexFormatWriter: Должен принимать строки из Paimon (InternalRow), накапливать их в векторы/пачки (batches) и сбрасывать на диск, используя API Vortex, сохраняя при этом метаданные (min/max значения для статистики).
   * VortexFormatReader: Должен читать векторы из файлов Vortex и проецировать их обратно в формат, понятный Paimon или движку исполнения (Flink/Spark).
 * Оптимизация Pushdown: Самая сложная и важная часть реализации — проброс фильтров (Predicate Pushdown) и выборки колонок (Projection Pushdown) из Paimon напрямую в сканер Vortex, чтобы читать с диска только нужные векторы.
