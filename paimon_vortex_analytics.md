Apache Paimon + Vortex: Векторизованный Real-time Lakehouse на базе Apache Arrow

Интеграция Apache Paimon и Vortex через призму Apache Arrow — это архитектура, которая переносит концепцию Zero-copy из оперативной памяти прямо на уровень дискового хранилища. Учитывая успешный опыт внедрения формата Lance в экосистему Paimon, использование Arrow в качестве "моста" для Vortex выглядит максимально логичным и технически элегантным решением.

Какие проблемы решает связка через Arrow?

Классические форматы (Parquet, ORC) требуют дорогостоящей десериализации: данные читаются с диска, распаковываются, прогоняются через словари и только потом превращаются в объекты Java или внутренние форматы движка.
Использование Arrow как транзитного формата решает эти проблемы кардинально:
* Истинный Zero-copy: Vortex спроектирован для мгновенной конвертации в Arrow RecordBatch. Данные поднимаются с диска и сразу же, без преобразований байтов, готовы к аналитическому сканированию.
* Векторизованное исполнение (Vectorized Execution): Аналитические движки (например, Flink в batch-режиме или Spark) умеют работать с Arrow напрямую, обрабатывая целые колонки (векторы) за один такт процессора (SIMD), а не по одной строке (Row-by-Row).
* Обход ограничений JVM: Arrow хранит данные вне кучи (off-heap memory), что радикально снижает нагрузку на Garbage Collector в Java-процессах Paimon.
* Использование Vortex: он позволяет передавать данные из файлового хранилища Paimon в аналитический движок практически без накладных расходов на CPU (zero-copy) и с максимальным использованием SIMD-инструкций процессора, а так же, потенциально с GPU оптимизациями.
  
На каком этапе используется Vortex (и где тут Arrow)?

  В архитектуре LSM-дерева Paimon формат Vortex выступает исключительно как физический слой хранения (Storage Layer) для Data Files. Arrow при этом является in-memory представлением этих данных.
* MemTable Flush: Данные накапливаются в памяти. При сбросе на диск VortexWriter берет внутреннее строковое представление Paimon, батчит его в Arrow-векторы и записывает на диск в формате Vortex.
* Чтение (Querying & Compaction): Это самый важный этап. При аналитическом запросе или фоновом уплотнении файлов (compaction), VortexReader читает блоки с диска и выдает их сразу в виде Arrow-векторов (в Paimon это абстракция ColumnVector или ColumnarRowData).
  Журналы упреждающей записи (WAL / Changelogs) при этом остаются в строковых форматах (например, Avro), так как Arrow и Vortex оптимизированы для аналитического (колончатого) чтения, а не для записи единичных транзакций.

  Архитектурная реализация (по аналогии с Lance)

  Чтобы внедрить Vortex в Paimon через Arrow, архитектура абстракций должна повторять паттерны векторизованного чтения:
* VortexFileFormat: Главный класс фабрики, который регистрируется в Paimon и указывает, что формат поддерживает векторизованное чтение.
* ArrowVortexReader: Компонент, который интегрируется с C++/Rust ядром Vortex (через JNI). Он мапит файловые блоки Vortex напрямую в Arrow-буферы (off-heap) и оборачивает их в Paimon VectorizedColumnBatch.
* Pushdown-оптимизации: Как и в случае с Lance, критически важно реализовать передачу фильтров (Predicate Pushdown) на уровень самого Vortex. Arrow позволяет вычислять маски фильтрации (Bitmaps) прямо по векторам, считывая с диска только те блоки данных, которые реально нужны запросу.

> Научный потенциал и бенчмарки: Архитектура идеальна для проведения производительных тестов. Для качественного сравнения стоит замерить скорость выполнения агрегационных SQL-запросов (через Flink SQL) на датасетах в Paimon(Vortex+Arrow) против Paimon(Parquet). Ожидаемый результат — кратное снижение задержек при чтении за счет отсутствия десериализации.
